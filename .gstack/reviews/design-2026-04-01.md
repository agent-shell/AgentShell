## Problem Statement

运维/DevOps 工程师每天管理 20-200 台服务器，主力工具是 SecureCRT/XShell。这类工具功能完整（SSH 会话管理、SFTP、rz/sz 文件传输、多窗口）但 AI 化程度接近零。

另一边，Warp/Wave 等 AI 终端针对的是本地 shell 开发者，不是 SSH 密集型运维场景，且 Warp 是闭源云端方案，数据不透明。

**AgentShell 的定位**：SecureCRT/XShell 的功能完整性 + 开源 AI Agent（审批模式 + 自动模式）+ 数据本地化。

---

## 核心用户

**主要用户**：管理多台服务器的运维/SRE/DevOps 工程师
- 每天活在 SSH 里，SecureCRT/XShell 是主力
- 使用 rz/sz 传文件（中国运维场景刚需）
- 需要跳板机、端口转发、多会话并行
- 对云端数据传输敏感，倾向本地/私有化部署
- 愿意为效率买单，但需要可信任的工具

**次要用户**：需要 SSH 的后端开发者、学生、开源贡献者

## 什么让这个产品"酷"

1. **rz/sz 是个隐藏信号** — 支持 rz/sz 意味着"这工具真的懂运维"，其他 AI 终端全部不支持
2. **AI 知道你连的是哪台服务器** — 不是对本地 shell 聊天，而是带着服务器上下文（hostname、当前目录、最近运行的命令、SFTP 文件结构）做决策
3. **双模式 AI** — 默认审批（每步可控）+ 自动模式（"帮我部署一套 k8s"，AI 执行完整流程）

## 功能范围

### MVP (v1) — GUI + SSH + AI 审批模式

**SSH 连接管理**
- 连接配置文件（host/port/user/auth 方法）
- 支持密码、公钥（PEM/OpenSSH 格式）、SSH Agent 认证
- 跳板机（Jump Host）链式连接
- 会话标签页（多 SSH 同时打开）
- 本地 Shell 标签页

**终端功能**
- xterm.js 渲染，支持颜色、Unicode、全屏程序（vim/htop）
- 终端分屏（水平/垂直）
- 快速搜索（Ctrl+F）
- URL 可点击
- 窗口 resize → PTY 自动 reflow

**文件传输**
- SFTP 文件浏览器（左侧面板或独立标签）
- 文件拖拽上传/下载
- **rz/sz (Zmodem) 协议支持** — 运维刚需，竞品全部缺失
- 进度条显示

**AI Agent（审批模式）**
- 右侧面板：AI 聊天，读取当前会话的终端上下文（最近 200 行）
- 提议命令 → CommandApproval 卡片（Run / Edit / Dismiss）
- 风险分级：safe（绿）/ caution（黄）/ destructive（红 + 二次确认）
- 多 AI 后端：Claude API / Ollama 本地 / OpenAI 兼容
- 审计日志：所有执行命令记录 timestamp + session + 命令内容

### v2 — AI 自动执行模式

**自动化工作流**
- 用户描述高层任务（"帮我在这三台机器上部署 k8s"）
- AI 生成完整执行计划（多步骤，跨多个会话）
- 用户批准计划后，AI 自动按序执行
- 每步执行前可暂停，遇到错误自动尝试修复或暂停等待人工介入
- 执行日志实时展示

**Session Context Memory**
- 跨会话记忆：上次连这台机器做了什么
- 服务器画像：系统版本、常用工具、典型问题

### rz/sz 实现方案

Zmodem 是 1980 年代的文件传输协议，基于字节流。实现方式：
1. 在 Rust 端拦截 PTY 输出流，检测 Zmodem 握手序列（`**\x18B0100...`）
2. 检测到后切换到 Zmodem 处理模式，暂停正常终端输出
3. 实现 Zmodem 接收（rz）和发送（sz）状态机
4. 完成后恢复正常终端模式，前端显示传输进度

参考：`zmodem-rs` crate 或参考 `lrzsz` 实现。

### AI 上下文架构

```
SSH 会话 PTY 输出
    │
    ▼
scrollback_buffer: VecDeque<u8>  (Rust, 环形缓冲 ~10000 行)
    │
    ▼ get_terminal_context() IPC 命令
    │  strip ANSI escapes
    ▼
contextBuilder.ts
  "连接: prod-db-01 (root@10.0.0.1)"
  "近期输出 (200 行):\n..."
    │
    ▼
Claude API / Ollama / OpenAI-compat (streaming)
    │
    ├── plain text → AgentMessage 渲染
    └── tool_use: propose_command → CommandApproval 卡片
```

## 安全模型

**AI 命令提议安全**
- 终端上下文在发送给 AI 之前，先 strip ANSI 转义序列（strip-ansi-escapes crate），防止 ANSI 注入攻击
- AI 响应中的 tool_use 块必须包含 `risk_level` 字段，前端根据风险等级展示不同 UI
- `destructive` 级命令需要用户二次勾选确认
- 命令内容在前端展示前做 HTML 转义，防止 XSS

**风险分级规则（MVP 内置规则集）**
| 级别 | 判断依据 | 示例 |
|------|---------|------|
| `safe` | 只读操作、无副作用 | `ls`, `cat`, `ps`, `df -h` |
| `caution` | 写操作但可逆 | `mkdir`, `touch`, `chmod`, `systemctl restart` |
| `destructive` | 不可逆或影响范围大 | `rm -rf`, `DROP TABLE`, `kill -9`, `format`, `>` 覆盖重要文件 |

AI 做初始分级，用户可手动调级。规则集放在配置文件中，可用户自定义。

**审计日志**
- 每次执行命令写入 `~/.agentshell/audit.log`（append-only JSON Lines）
- 字段：timestamp、session_id、hostname、command、risk_level、approved_by（"user"）
- 日志文件权限 600，只有当前用户可读
- v2 支持导出审计报告

**CommandApproval 状态机**
```
[AI 提议] → proposed
    ├── [用户点 Edit] → editing（用户修改命令文本）
    │       └── [用户点 Run] → approved（记录"user-edited"）→ executed → logged
    ├── [用户点 Run] → approved → executed → logged
    └── [用户点 Dismiss] → dismissed（记录 dismissed，不执行）
```
已修改的命令重新计算风险等级（AI 重新分析修改后文本）。Dismiss 操作同样记录到审计日志。

**离线可用性**
SSH / SFTP / rz-sz / 本地 Shell 在无网络、无 AI 后端的情况下**完全可用**。AI Agent 面板在无后端时显示"未配置 AI 后端"提示，不影响终端功能。

## rz/sz (Zmodem) 实现规格

**协议检测**
- 在 Rust PTY 输出流中扫描 Zmodem 握手序列：`**\x18B0100` (rz 启动) 和 `**\x18B0000` (sz 启动)
- 检测到后 PTY 切换到 Zmodem 模式，暂停正常 xterm.js 渲染
- 前端显示文件传输进度条覆盖层

**状态机**
```
NORMAL → [检测握手] → ZMODEM_HANDSHAKE
ZMODEM_HANDSHAKE → [确认] → ZMODEM_TRANSFER
ZMODEM_TRANSFER → [完成] → NORMAL
ZMODEM_TRANSFER → [中断/超时] → ZMODEM_ERROR → [用户确认] → NORMAL
```

**错误恢复**
- 传输中断（网络断开/PTY 关闭）：写入已接收的部分文件 + `.partial` 后缀，通知用户
- 重试：重新运行 `rz` 命令，协议层支持断点续传（Zmodem CAN-AT 机制）
- 超时：30 秒无数据自动中止，清理状态

**平台差异**
- macOS/Linux：ConPTY 不涉及，字节流直接拦截，实现可靠
- Windows：ConPTY 的字节流有转换层，Zmodem 字节检测需要额外验证
- **v1 范围**：macOS + Linux 支持 rz/sz，Windows 标注"Beta/实验性"，v1.1 修复

**库选择**：使用 `lrzsz` 参考实现自写 Rust 状态机，或评估 `zmodem2` crate。不引入外部二进制（lrzsz 命令行工具）。

## 技术细节补充

**PTY 默认尺寸**：初始 80×24，前端挂载后立即触发 ResizeObserver → FitAddon.fit() → resize_pty IPC，适配实际窗口大小。

**Scrollback 缓冲**：`VecDeque<u8>`，最大 10MB（约 50,000 行典型输出）。超出时丢弃最旧的行（FIFO）。AI 上下文取最近 200 行（可配置 50-500 行）。ANSI 序列在发送给 AI 前 strip，但原始字节保留在缓冲区供终端渲染。

**AI 流式架构**
```
Claude API (streaming)
    │  SSE chunks
    ▼
streamParser.ts
    ├── text_delta → 追加到 AgentMessage（实时渲染）
    ├── input_json_delta → 累积 tool_use 参数
    └── message_stop → 如果有完整 tool_use → 生成 CommandProposal

错误处理：
- 连接中断 → 显示"响应被中断"，已收到部分内容保留
- 超时（30s 无响应）→ 自动重试一次，再失败提示用户
- partial tool_use → 等待 message_stop 后再解析，不中途渲染
- **SSH 会话断开（AI 流式响应中途）**：立即丢弃全部 partial tool_use，清除当前会话的所有 CommandApproval 卡片，显示"连接断开，响应已取消"+ 快速重连按钮
```

**russh 认证覆盖**（需在实现前验证）
- [x] 密码认证
- [x] PEM 格式私钥（RSA 2048+）
- [x] OpenSSH 格式私钥（ed25519、ecdsa）
- [x] SSH Agent 转发（SSH_AUTH_SOCK / Windows Pageant 命名管道）

## CI/CD 已知问题（v1）

- **macOS 代码签名**：需要 Apple Developer 证书（$99/年）。v1 可先发未签名版本，提示用户 "右键 → 打开" 绕过 Gatekeeper。v1.1 加入签名。
- **Windows .msi 签名**：未签名会触发 SmartScreen 警告。v1 先发未签名，README 说明。
- **Linux AppImage**：无签名问题，AppImage 格式天然可执行。
- **GitHub Actions Tauri 构建**：使用 `tauri-apps/tauri-action@v0` 官方 Action，三平台 matrix。已知坑：Ubuntu 需要额外安装 `libwebkit2gtk-4.1-dev`。
