## Vision

### 10x Check

AgentShell 的 10x 版本不是"更好的终端"——它是**运维团队的 AI 大脑**。想象一个场景：你的 SRE 团队有 200 台服务器，分布在三个机房。AgentShell 知道每台服务器的状态、历史变更、典型配置。当一台服务器报警，AI 自动跨 5 台机器拉日志、关联告警、生成根因分析、提议修复步骤——你只需要点 "批准"。这不是聊天机器人加终端，这是**有记忆、有判断力的运维自动化平台**，以终端 UI 为控制平面。

## Accepted Scope (v1 范围更新)

**原有 v1 功能（保持不变）：**
- SSH 连接管理（密码/公钥/SSH Agent/跳板机）
- 多 Tab + 本地 Shell Tab
- xterm.js 终端（分屏、搜索、URL 点击）
- SFTP 文件浏览器 + 拖拽传输
- rz/sz Zmodem 支持
- AI Agent 面板（Claude/Ollama/OpenAI-compat，审批模式）
- 审计日志

**新增入 v1 范围：**
- Tab 栏服务器健康指标（SSH 定期采集 CPU/内存，绿/黄/红圆点）
- 会话录制+回放（asciinema 格式，存 `~/.agentshell/recordings/`）
- 服务器标签分组（ConnectionProfile.tags 字段，侧边栏按标签树形展示）
- 命令历史快速搜索（shell 历史 + SQLite FTS5 全文检索，模糊 Ctrl+R；v1.1 升级真正的向量语义搜索）

## Architecture Landmines（CEO 视角）

以下是我在 SELECTIVE EXPANSION 评审中发现的最高风险点：

### L1: 命令历史搜索实现（已决定）
**v1 用 SQLite FTS5 全文检索**（无向量，无 AI 调用）。实现：每次命令执行后写入 SQLite `commands` 表（session_id, hostname, command, timestamp, exit_code），FTS5 虚表提供全文模糊搜索。前端 Ctrl+R 弹出搜索框，实时查询。v1.1 在此基础上加入向量嵌入升级为真正的语义搜索。此决定已锁定，无需进一步讨论。

### L2: 健康指标采集实现（已决定）
**v1 用 `uptime` 命令，通过独立 SSH exec channel（非主 shell）采集，默认每 60s 一次，可配置范围 10-300s。**
- 颜色规则：load_1m < cpu_count → 绿；cpu_count ≤ load_1m < 2×cpu_count → 黄；load_1m ≥ 2×cpu_count → 红
- 独立 exec channel 意味着不干扰主 shell，但每次采集建立一个短暂 channel（russh exec channel，无新 TCP 连接）
- 100 台机器 60s 采集一次 = 1.67 次/秒，负载极低
- Zmodem 模式下健康采集暂停（参见 L4）

### L3: 会话录制 vs 存储增长
策略：不硬截断，软警告 + 自动归档。
- 录制达到 100MB 时：状态栏黄色警告"录制文件已超 100MB"
- 录制达到 500MB 时：自动停止录制，通知用户
- 自动归档：30 天未访问的录制文件移至 `~/.agentshell/recordings/archive/`
- 回放：v1 内嵌简单 asciinema-player（WebAssembly 版，无需外部工具），或调用系统 `asciinema play` 命令（如已安装）

### L4: Zmodem + 健康采集 冲突
Zmodem 传输期间 PTY 切换到 Zmodem 模式（暂停正常输出）。如果此时健康采集 goroutine 尝试读取 SSH channel 输出，会干扰 Zmodem 握手。**建议**：session_manager.rs 维护一个 `pty_mode: PtyMode { Normal, Zmodem }` 状态，健康采集 goroutine 检查此状态，Zmodem 模式下暂停采集。
